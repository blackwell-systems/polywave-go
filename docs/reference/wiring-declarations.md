# Wiring Declarations Reference

Wiring declarations express a binding contract: a symbol defined by one agent
must be called from a specific file owned by another agent in the same wave.
They are written by Scout in the IMPL manifest's `wiring:` block and enforced
by three layers of tooling before and after wave execution.

---

## WiringDeclaration Schema

`wiring:` is a top-level YAML sequence in the IMPL manifest. Each entry is a
`WiringDeclaration`.

```yaml
wiring:
  - symbol: RegisterRoutes
    defined_in: pkg/api/routes.go
    must_be_called_from: pkg/server/server.go
    agent: B
    wave: 1
    integration_pattern: call
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `symbol` | string | yes | Exported symbol name (function, method, type). Must match the exact identifier as it appears in source. |
| `defined_in` | string | yes | Repo-relative path to the file where `symbol` is defined. |
| `must_be_called_from` | string | yes | Repo-relative path to the file that must call or reference `symbol`. This file must be in `agent`'s `file_ownership` for this wave. |
| `agent` | string | yes | Agent ID responsible for wiring `symbol` into `must_be_called_from`. Must match an agent ID in `file_ownership`. |
| `wave` | int | yes | Wave number. Only wiring declarations matching the current wave are injected into agent briefs. |
| `integration_pattern` | string | no | Hint describing how the symbol is wired in. See [integration_pattern values](#integration_pattern-values). |

The `wiring_declaration` key set is enforced by `DetectUnknownKeys` (V013). Any
unrecognized key is a validation error.

### integration_pattern values

`integration_pattern` is informational — it appears in the agent brief table
and is not validated as an enum. Conventions used in the codebase:

| Value | Meaning |
|-------|---------|
| `call` | `symbol` is invoked as a function call. Default for most functions. Auto-generated by `polywave-tools detect-wiring`. |
| `append` | `symbol` is appended to a slice or registry (e.g., route slice, middleware chain). |
| `register` | `symbol` is registered with a dispatcher, mux, or service container. |
| `prop-pass` | `symbol` is passed as a React prop (TSX only). |

Empty `integration_pattern` is valid. The field is omitted from YAML when
empty (`omitempty`).

## How Wiring Blocks Prevent E35 Gaps

**E35** is the failure mode where a function defined by one agent is called
from a file not owned by that agent, creating a race condition: the defining
agent may change the signature while the call site is frozen at the old
signature, producing a build failure after merge.

A wiring declaration makes the calling obligation explicit so that:

1. The agent brief includes a `## Wiring Obligations (E35)` section listing
   every symbol the agent must wire (Layer 3C).
2. The `must_be_called_from` file is validated against the agent's
   `file_ownership` before the wave starts (Layer 3A).
3. After merge, `validate-integration --wiring` confirms the symbol appears as
   a call in `must_be_called_from` (Layer 3B).

Without a wiring declaration, a same-package call gap is only surfaced by
`DetectE35Gaps` (used by `polywave-tools pre-wave-validate`), which identifies
functions defined in one agent's files but called from files owned by a
different agent in the same wave.

## Enforcement Layers

### Layer 3A — Pre-flight ownership check (prepare-wave)

`CheckWiringOwnership(manifest, waveNum)` verifies that each declaration's
`must_be_called_from` file appears in the named agent's `file_ownership` for
that wave. Violations are returned as strings of the form:

```
wiring entry for "<symbol>": must_be_called_from "<file>" is not in agent <ID> file_ownership (wave <N>)
```

A violation means the Scout assigned the wiring obligation to an agent who
does not own the caller file. Either fix the agent assignment or add a
`file_ownership` entry for the caller file.

### Layer 3B — Post-merge symbol check (validate-integration --wiring)

`ValidateWiringDeclarations(manifest, repoPath)` checks each declaration after
wave agents complete and their branches are merged. For each entry it opens
`must_be_called_from` and searches for `symbol`:

- `.go` files: Go AST walk for `CallExpr` nodes matching the symbol as a bare
  identifier (`Symbol(...)`) or selector (`pkg.Symbol(...)`).
- `.go` files that fail AST parsing: falls back to line scan.
- TSX/TS files: prop-aware regex detection that skips comment lines and type
  definitions.
- All other file types: substring line scan.

If the file does not exist or cannot be read, that is a gap. If the symbol is
not found, that is a gap. All gaps have `severity: "error"`.

Results are returned as `WiringValidationData` and persisted to the manifest
under `wiring_validation_reports.<waveKey>` by `AppendWiringReport`.

### Layer 3C — Agent brief injection (prepare-wave)

`InjectWiringInstructions(doc, agentID, waveNum)` (also exposed as
`FormatWiringBriefSection`) filters declarations by both agent ID and wave
number and renders a markdown table injected into `.polywave-agent-brief.md`:

```markdown
## Wiring Obligations (E35)

You must wire the following exports into existing aggregation points.
These declarations are specified in the IMPL doc `wiring:` block and will be
validated post-merge by `validate-integration` (severity: error if missing).

| Symbol | Defined In | Must Be Called From | Pattern |
|--------|------------|---------------------|--------|
| RegisterRoutes | pkg/api/routes.go | pkg/server/server.go | call |
```

An agent with no wiring obligations for the current wave receives no
`## Wiring Obligations` section.

## IntegrationReport Structure

`IntegrationReport` is produced by `ValidateIntegration` (E25) via
`polywave-tools validate-integration` after wave agents complete. It is distinct from
`WiringValidationData` (E35 Layer 3B): E25 detects heuristic integration gaps
(exports with no call sites in the codebase), while E35 Layer 3B checks
explicit wiring declarations.

```go
type IntegrationReport struct {
    Wave    int              // wave number
    Gaps    []IntegrationGap // detected unconnected exports
    Valid   bool             // true if no gaps detected
    Summary string           // human-readable summary
}
```

### IntegrationGap fields

| Field | Type | Description |
|-------|------|-------------|
| `export_name` | string | Name of the exported symbol with no detected call sites. |
| `file_path` | string | File containing the export. |
| `agent_id` | string | Agent that produced the file. |
| `category` | string | `function_call`, `type_usage`, `field_init`, `prop_pass`, or `none`. |
| `severity` | string | `error`, `warning`, or `info`. Filtered by `integration_gap_severity_threshold`. |
| `reason` | string | Why the gap was detected. |
| `suggested_fix` | string | Heuristic suggestion for where to add the call site. |
| `search_results` | []string | Files identified as likely callers via import graph or grep. |

Integration reports are persisted to the manifest under
`integration_reports.<waveKey>` and consumed by the Integration Agent (E26).

`ValidateIntegration` runs at step 4 of `FinalizeWave`. It is informational
and non-blocking — gaps do not halt the merge pipeline.

## E26 Integration Agent Workflow

When `IntegrationReport.Valid` is false (gaps exist), `RunIntegrationAgent`
(E26) launches an LLM agent to wire the gaps into caller files. Two
preconditions must hold:

1. **E26-P1**: `integration_reports` must be persisted (E25 must have run).
   If absent, the engine logs a warning and continues with the caller-supplied
   report.
2. **E26-P2**: `integration_connectors` must be non-empty. The integration
   agent is only permitted to modify files listed there.

```yaml
integration_connectors:
  - file: pkg/server/server.go
    reason: Aggregation point for new route registrations
  - file: cmd/polywave/main.go
    reason: Top-level wiring of engine subsystems
```

The integration agent runs in the main repo directory (not a worktree) so it
sees the merged result of all wave agents. Its file-write scope is constrained
to `integration_connectors` entries via `BuildIntegratorConstraints`.

### E27 — Integration wave type

Setting `type: integration` on a wave marks it as a wiring-only wave: no
worktrees are created for its agents, and agents operate directly on the
merged branch. This is used when wiring work is decomposed into a dedicated
wave rather than handled by the E26 integration agent.

```yaml
waves:
  - number: 2
    type: integration
    agents:
      - id: A
        task: Wire RegisterRoutes into server.go
        files: [pkg/server/server.go]
```

## WiringValidationData Structure

`WiringValidationData` is returned by `ValidateWiringDeclarations` and
persisted to `wiring_validation_reports.<waveKey>` in the manifest.

```go
type WiringValidationData struct {
    Gaps    []WiringGap // declarations where symbol was not found
    Valid   bool        // true if all declarations are satisfied
    Summary string      // e.g. "all 3 wiring declarations satisfied"
}

type WiringGap struct {
    Declaration WiringDeclaration // the unsatisfied declaration
    Reason      string            // why the symbol was not found
    Severity    string            // always "error"
}
```

`Summary` format:
- All satisfied: `"all <N> wiring declarations satisfied"`
- Gaps found: `"<N> wiring gap(s) found in <M> declarations"`

## Wiring vs. Reassigning Files

Use a wiring declaration when an agent must call a function defined in a file
owned by a different agent in the same wave. Do not reassign the caller file
to the defining agent unless the file's primary purpose aligns with that
agent's task.

Wiring declarations are appropriate when:

- The caller file (`must_be_called_from`) already has a clear owner (e.g., a
  server setup file owned by an infrastructure agent) and the callee is a new
  symbol produced by a feature agent.
- The integration work is one or two call sites — not broad refactoring.
- You want Layer 3B enforcement to verify the wiring after merge.

Reassign the file when:

- The new agent's work is logically inseparable from the caller file's
  existing content.
- The wiring is so extensive that a different agent owning the file reduces
  coordination overhead across waves.
- The E35 gap would require the integration agent (E26) to understand deep
  context already held by the defining agent.

`polywave-tools detect-wiring <impl-path>` auto-generates candidate wiring
declarations by scanning agent task prompts for patterns like
`"calls \`FunctionName()\`"` and cross-referencing `file_ownership`. Output is
YAML in `wiring:` schema format; Scout should review before committing (~80%
pattern-match reliability).
